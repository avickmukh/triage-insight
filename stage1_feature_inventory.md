# TriageInsight Stage-1 AI Feature Inventory

**Report Date:** Mar 27, 2026

This document provides a complete, evidence-based inventory of the Stage-1 Semantic Intelligence features currently available and fully working in the `triage-insight` codebase as of commit `83d8caa`.

---

## High-Level Summary

The core Stage-1 pipeline is **fully implemented and operational**. When new feedback is created, a background job automatically generates an embedding, assigns the feedback to a semantic theme, and identifies potential duplicates. The UI has the necessary endpoints to display this information.

| Feature | Status | Description |
| :--- | :--- | :--- |
| **Embedding Generation** | ✅ **Fully Working** | Feedback text is converted into a 1536-dimension vector via OpenAI. |
| **Semantic Clustering** | ✅ **Fully Working** | New feedback is automatically assigned to the most similar existing theme. |
| **Automatic Theme Creation** | ✅ **Fully Working** | If no sufficiently similar theme exists, a new one is created from the feedback. |
| **Duplicate Candidate Generation** | ✅ **Fully Working** | Highly similar feedback items are identified and saved as `PENDING` suggestions. |
| **Related Feedback Linking** | ✅ **Fully Working** | Relatedness is exposed via theme membership and duplicate suggestions. |
| **Priority Scoring Trigger** | ✅ **Fully Working** | A job to calculate a CIQ priority score is enqueued on feedback creation. |
| **Manual Re-clustering** | ✅ **Fully Working** | An admin endpoint exists to trigger a full workspace re-clustering job. |
| **Semantic Search (User-Facing)** | ❌ **Not Implemented** | No API endpoint exists for users to perform vector-based semantic search. |

## Detailed Feature Breakdown

### 1. Embedding Generation

| Status | ✅ **Fully Working** |
| :--- | :--- |
| **Service** | `apps/api/src/ai/services/embedding.service.ts` |
| **Processor** | `apps/api/src/ai/processors/analysis.processor.ts` (Step 3) |
| **DB Field** | `Feedback.embedding` (`vector(1536)`) |

**How it works:**
1.  When feedback is created, an `ai-analysis` job is enqueued [1].
2.  The `AiAnalysisProcessor` consumes this job.
3.  It calls `EmbeddingService.generateEmbedding()`, which sends the feedback description to the OpenAI `text-embedding-3-small` model to get a 1536-dimension vector [2].
4.  The processor then persists this vector directly to the `embedding` field on the `Feedback` table using a raw SQL query with a `::vector` cast [3].

This entire flow is operational and correctly implemented.

### 2. Semantic Clustering & Automatic Theme Creation

| Status | ✅ **Fully Working** |
| :--- | :--- |
| **Service** | `apps/api/src/ai/services/theme-clustering.service.ts` |
| **Processor** | `apps/api/src/ai/processors/analysis.processor.ts` (Step 5) |
| **DB Table** | `Theme`, `ThemeFeedback` |

**How it works:**
1.  After generating the embedding, the `AiAnalysisProcessor` calls `ThemeClusteringService.assignFeedbackToTheme()` [3].
2.  This service performs a cosine similarity search (`<=>`) against all existing `Theme` embeddings in the same workspace.
3.  **If a similar theme is found** (similarity > 0.80), a new record is created in the `ThemeFeedback` table, linking the feedback to the existing theme [4].
4.  **If no similar theme is found**, a new `Theme` is created. The theme's title is generated by the `SummarizationService` (using GPT-4.1-mini), and its embedding is the same as the feedback's embedding. A `ThemeFeedback` record is then created to link them [4].

This allows feedback to be automatically grouped into semantically related clusters.

**Manual Re-clustering:**
An admin-only endpoint `POST /workspaces/:workspaceId/themes/recluster` exists to trigger a full, workspace-wide re-clustering job via the `ThemeClusteringProcessor` [5]. This is useful for re-evaluating all feedback against a new set of themes.

### 3. Duplicate Candidate Generation

| Status | ✅ **Fully Working** |
| :--- | :--- |
| **Service** | `apps/api/src/ai/services/duplicate-detection.service.ts` |
| **Processor** | `apps/api/src/ai/processors/analysis.processor.ts` (Step 4) |
| **DB Table** | `FeedbackDuplicateSuggestion` |

**How it works:**
1.  After embedding generation, the `AiAnalysisProcessor` calls `DuplicateDetectionService.generateSuggestions()` [3].
2.  This service performs a cosine similarity search against all other `Feedback` embeddings in the same workspace.
3.  For any feedback item with a similarity score > 0.90, a new record is created in the `FeedbackDuplicateSuggestion` table with a status of `PENDING` [6].
4.  This suggestion links the source and target feedback items and stores the similarity score.

**UI Integration:**
- The `GET /workspaces/:workspaceId/feedback/:id` endpoint includes `duplicateSuggestionsAsSource` in its response, so the UI can display pending suggestions on the feedback detail page [7].
- The `GET /workspaces/:workspaceId/feedback/:id/potential-duplicates` endpoint specifically returns these AI-generated suggestions [8].
- A dedicated controller at `apps/api/src/ai/controllers/duplicate-suggestions.controller.ts` provides endpoints for listing, accepting, and rejecting these suggestions, which facilitates building a full management UI [9].

### 4. Related Feedback Linking

| Status | ✅ **Fully Working** |
| :--- | :--- |
| **Mechanism** | Theme Membership & Duplicate Suggestions |

**How it works:**
The concept of "related feedback" is not implemented as a direct, arbitrary link between two feedback items. Instead, it is exposed to the user through two primary, fully-working mechanisms:

1.  **Theme Membership:** All feedback items assigned to the same semantic theme are, by definition, related. The UI can (and does) use the `ThemeFeedback` table to show all items within a theme.
2.  **Duplicate Suggestions:** The `FeedbackDuplicateSuggestion` table provides explicit links between highly similar items.

The `findOne` method in `feedback.service.ts` correctly includes both `themes` and `duplicateSuggestionsAsSource` in its response, providing all necessary data for a UI to display related items [7].

### 5. Priority Scoring Trigger

| Status | ✅ **Fully Working** |
| :--- | :--- |
| **Service** | `apps/api/src/ai/services/ciq.service.ts` |
| **Processor** | `apps/api/src/ai/processors/ciq-scoring.processor.ts` |
| **DB Field** | `Feedback.ciqScore`, `Theme.priorityScore` |

**How it works:**
1.  When feedback is created, in addition to the `ai-analysis` job, a `ciq-scoring` job is also enqueued with the type `FEEDBACK_SCORED` [1].
2.  The `CiqScoringProcessor` consumes this job and uses the `CiqService` and `CiqEngineService` to calculate a priority score based on various signals (this scoring logic itself is part of a separate, more complex feature set beyond Stage-1).
3.  The key point for Stage-1 is that the **trigger path is fully wired**. The job is enqueued, the processor is registered, and the basic persistence of the score to the `Feedback` and `Theme` tables is implemented [10].

### 6. Semantic Search (User-Facing)

| Status | ❌ **Not Implemented** |
| :--- | :--- |

While the backend has all the necessary components for semantic search (embeddings on every feedback item), there is **no user-facing API endpoint** to perform this action. The existing `GET /feedback?search=` endpoint performs a standard keyword-based `ILIKE` search [11].

**Implementation Gap:**
A new endpoint, such as `GET /feedback/semantic-search?q=`, would be required. This endpoint would:
1.  Take the user's query string `q`.
2.  Generate an embedding for the query using `EmbeddingService`.
3.  Perform a cosine similarity search against the `Feedback.embedding` column in the database.
4.  Return the most similar feedback items.

This is a feature gap, not a bug. The foundational data layer is ready for it to be built.
